# Ontology Modeling: Bi-Chat Orchestration Report

## Methodology Overview

This project employs a sequential two-chat orchestration methodology to extend existing ontologies and address specific Competency Questions (CQs). By decoupling design from execution, we maintain high logical consistency and clean code generation. This was to ensure a Human-In-The-Loop (HITL) approach was easier as it allowed to clarify and guide an Orchestrator chat, only allowing it to pass verified prompts to the Builder Chat

* Chat A - The Orchestrator: Acts as the strategic lead. It handles research, evaluates outputs from the builder, and performs "Prompt Engineering" to create high-fidelity instructions.
* Chat B - The Builder: Acts as the technical executor. It receives refined prompts from the Orchestrator to perform script generation and RDF construction without the "noise" of the brainstorming phase.

* For full Builder prompts generated at each phase, see: PROMPT.MD in the Git repository linked in the report.

## Phase 1: Mapping and Prompt Creation

Goal: Identify logical mappings between domain requirements and the existing ontology schema.

Orchestrator Prompt:
"Create a prompt based on the following framework which will be utilised by a different chat.

Scenario:
I am creating an ontology of the London underground.

Inputs:
- 2 Existing Ontologies
- 10 Competency Questions

Task:
Takes 2 existing ontologies which I give you, and uses the set of 10 CQs. Take the 10 CQs and extract classes, relations and properties from it. Then map it to the 2 ontologies and try to use existing classes as much as possible but ensure each existing ontology is extended by at least 2 subclasses and subproperties each. Lastly from those extracted classes and relations create 10 more CQs which will later be translated into SPARQL queries.

Goal:
The goal is to extract the classes and relations from the CQs and generate 10 more CQs.

Output:
- A mapping of Classes, relations and properties from the CQs which I need to have in order to extend the imported ontologies.
- 10 new CQs."


## Phase 2: Ontology Creation & Continuation

Goal: Develop the core Python scripts required to programmatically generate the expanded ontology.

Orchestrator Prompt:
"This is the response from the builder:

[Insert Response here]

Generate a prompt to task the builder to utilise its previous response to generate the ontology_builder using rdflib."


## Phase 3: Verification and Correction (1)

Goal: Validate the generated ontology against CQs and fix logical inconsistencies or syntax errors.

Orchestrator Prompt:
"I have attached the generated script and the .ttl which it generated. Evaluate it against the existing 20 competency questions and give another prompt to generate 10 additional validation competency questions for real-world application and corrections if required.

Attachments:
- ontology_builder_initial.py (Script provided by builder chat)
- tbox_ontology.ttl (file produced by ontology_builder)"


## Phase 4: Verification and Correction (2)

Goal: Ensure alignment with external ontologies and competency questions.

Orchestrator Prompt:
"Verify the fixed ontology_builder.py and its declared classes, properties and relationships are valid.

Attachments:
- ontology_builder.py (updated version)
- tbox_ontology.ttl (produced by updated script)

Then provide me with a prompt, which identifies the issues and how to fix them."


## Phase 5: Documentation Cleanup

Goal: Finalise the knowledge engineering documentation and ensure all scripts and TTL files are formatted.

Orchestrator Prompt:
"Provide me with a prompt which allows the generation chat to include labels and comments for every class, property and relationship it defined. It shouldn't change any logic or relation but just add the metadata such as comments."


# RAG Implementation

Goal: Implement a RAG strategy to resolve some of the incompleteness found

Prompt used:
"Here is a pdf explaining my project task, I need to implement the RAG strategy task. I'd like to implement RAG for the following two gaps: Accessibility assessments (TfL API + LLM formatting) and Missing fare zones (TfL API + LLM disambiguation). I have attached 3 Turtle (.ttl) files below. Please help me generate a python script that uses RAG to resolve these two gaps in our ontology."

Note: Two additional LLM prompts were produced to fill the identified gaps (fare zone disambiguation and accessibility assessment). See: PROMPT.MD

# Pipeline LLM

Goal: Extract information and data from an LLM

"You are an information extraction system for a London Underground knowledge graph.

Given a disruption description, extract the following fields and return them as JSON:

- "incident_cause": the root cause of the disruption as a short noun phrase
(e.g. "signal failure", "track defect", "person on track", "engineering works",
"staff shortage", "faulty train"). Use null if the cause is not stated.

- "incident_type": classify the event into exactly one of these categories:
"planned_maintenance", "signal_failure", "person_on_track", "train_fault",
"station_closure", "other_unplanned". Use null if it cannot be determined.

- "is_planned": true if this is scheduled or planned maintenance, false if it is
an unplanned disruption. Use null if unknown.

Return ONLY a valid JSON object with these three keys. No explanation."
